# CI/CD：从持续集成到持续交付

## 背景与目标

在软件研发中，代码提交频率越来越高，系统依赖越来越复杂。  
如果仍依赖手工构建、手工测试和手工发布，交付效率和质量都会迅速下降。

CI/CD 的目标是把“构建、测试、发布”标准化、自动化，使团队能够：

- 更快交付业务价值；
- 更早发现缺陷和集成风险；
- 更稳定地发布版本并可快速回滚。

## 核心概念

### CI（持续集成，Continuous Integration）

持续集成强调开发者频繁提交代码，并在每次提交后自动触发构建和测试，尽早暴露问题。  
其核心是“**小步提交、快速反馈**”。

### CD（持续交付/持续部署，Continuous Delivery/Deployment）

CD 在行业中有两种常见含义：

- **持续交付（Continuous Delivery）**：代码通过自动化验证后，随时可以发布到生产环境，但最终发布动作可人工触发；
- **持续部署（Continuous Deployment）**：通过验证后自动发布到生产环境，不需要人工审批。

> 简单理解：持续交付强调“随时可发”，持续部署强调“自动就发”。

## 为什么需要 CI/CD

- **降低集成风险**：避免长时间分支开发导致“最后一天大爆炸”；
- **提升交付效率**：减少重复手工操作和等待时间；
- **提高质量稳定性**：通过自动化测试和质量门禁前置发现问题；
- **增强可追溯性**：每次变更都有流水线记录和产物版本；
- **支持快速回滚**：发布失败可快速切回稳定版本。

## CI/CD 标准流程

一个典型流水线通常包含以下阶段：

1. **代码提交**：开发者提交到主干或合并请求分支；
2. **编译与构建**：安装依赖、编译代码、生成构建产物；
3. **静态检查**：代码规范扫描、安全扫描、依赖漏洞检查；
4. **自动化测试**：单元测试、接口测试、必要时执行集成测试；
5. **打包与制品管理**：构建镜像或包，上传制品仓库；
6. **部署到测试/预发环境**：自动部署并执行验收用例；
7. **发布到生产环境**：可采用蓝绿/灰度/金丝雀策略；
8. **监控与回滚**：观察关键指标，异常时自动或手工回滚。

## 关键实践设计

### 分支与合并策略

- 推荐主干开发（Trunk-Based）或短周期功能分支；
- 合并请求必须通过流水线和代码评审；
- 禁止绕过质量门禁直接合并到主干。

### 质量门禁（Quality Gate）

至少应包含以下门禁：

- 构建成功；
- 单元测试通过（含最低覆盖率要求）；
- 核心静态规则通过；
- 高危安全漏洞为零；
- 关键服务冒烟测试通过。

### 制品与版本管理

- 制品必须不可变（immutable）；
- 版本号可追溯到提交哈希（commit hash）；
- “同一版本包”在不同环境复用，避免环境漂移。

### 发布策略

- 低风险系统可滚动发布；
- 核心系统建议灰度发布或金丝雀发布；
- 发布必须有明确回滚开关和回滚步骤。

## 常见工具链（示例）

- 代码托管：GitHub / GitLab / Gitee；
- 流水线：GitHub Actions / GitLab CI / Jenkins；
- 制品仓库：Nexus / Harbor / Artifactory；
- 容器与编排：Docker / Kubernetes；
- 监控告警：Prometheus / Grafana / Alertmanager。

> 工具不是重点，关键是流程标准化和团队执行一致性。

## 团队落地建议

### 从小步开始

先实现最小可用流水线：**构建 + 单测 + 制品归档**，再逐步引入安全扫描、自动部署和灰度发布。

### 明确角色职责

- 开发：保证代码质量与测试可运行；
- 测试：维护自动化用例质量；
- 运维/平台：维护流水线稳定性和部署能力；
- 管理者：定义发布节奏、质量红线和应急流程。

### 建立度量指标

建议持续跟踪以下指标：

- 部署频率（Deployment Frequency）；
- 变更前置时间（Lead Time for Changes）；
- 变更失败率（Change Failure Rate）；
- 平均恢复时间（MTTR）。

## 常见误区

- 把 CI/CD 当作“装个 Jenkins 就完事”；
- 流水线阶段过多过重，反馈周期过长；
- 只追求速度，不设置质量门禁；
- 测试不稳定导致流水线频繁“假失败”；
- 缺少回滚预案，发布失败时手忙脚乱。

## 小结

- CI 解决的是“持续集成与快速反馈”；
- CD 解决的是“稳定交付与发布自动化”；
- 真正有效的 CI/CD 不是工具堆砌，而是**流程、质量和组织协同能力**。

## 落地检查清单（Checklist）

- [ ] 是否建立了统一代码提交与合并策略；
- [ ] 是否具备基础流水线（构建、测试、制品）；
- [ ] 是否配置了明确质量门禁；
- [ ] 是否支持一键部署与一键回滚；
- [ ] 是否定义并持续跟踪关键效能指标；
- [ ] 是否定期复盘流水线效率与失败原因。
