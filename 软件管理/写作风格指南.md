# 软件管理写作风格指南

本指南用于统一本项目技术文章的结构、语气和排版，确保内容可读、可复用、可维护。

## 一、写作目标

- 面向初中级读者，降低理解门槛；
- 强调“概念清晰 + 可执行实践”；
- 保持章节之间风格一致，便于连续阅读。

## 二、统一结构（每篇建议）

1. `# 标题`
2. `## 背景与目标`
3. `## 核心概念`
4. `## 方法/流程`
5. `## 示例或图解`
6. `## 常见误区`
7. `## 小结`

> 不必机械照搬，但建议至少包含“背景、概念、方法、小结”四部分。

## 三、标题与段落规则

- 标题层级不跳级：`#` -> `##` -> `###`；
- 每段只表达一个重点，控制在 2-4 行；
- 长句拆短，优先使用“结论 + 原因 + 示例”句式；
- 相邻段落有逻辑连接词，如“因此、此外、例如、最后”。

## 四、术语与语言规则

- 术语首次出现采用“中文（English）”，例如：冲刺（Sprint）；
- 同一术语全文统一写法，不混用同义词；
- 用词客观准确，避免口语化和模糊词；
- 中英文之间留空格，例如：`Scrum 团队`、`MVP 版本`。

## 五、列表与步骤规则

- 枚举使用 `-`；
- 有先后顺序的操作使用数字列表 `1. 2. 3.`；
- 每个列表项尽量保持同一语法结构，控制在一行到两行；
- 同层级列表建议 3-6 条，避免过长。

## 六、图片与图注规则

- 图片统一使用相对路径，如 `![说明](../images/xxx.png)`；
- 图片放在相关段落后，避免“图文分离”；
- 需要解释时补一行图注：`*图 X：说明文字。*`；
- 图注用一句话说明“图展示了什么”和“读者该关注什么”。

## 七、格式细节清单（发布前自检）

- 是否存在重复段落或重复要点；
- 是否有错别字、术语不一致；
- 是否有过长段落（超过 5 行）；
- 是否所有图片都能在预览和 GitHub 正常显示；
- 是否有明确的小结，帮助读者快速回顾。

## 八、示例模板

```markdown
# 章节标题

## 背景与目标
一句话说明本章解决的问题。

## 核心概念
概念 A：定义。
概念 B：定义。

## 方法/流程
1. 第一步；
2. 第二步；
3. 第三步。

## 常见误区
- 误区 1；
- 误区 2。

## 小结
- 要点 1；
- 要点 2；
- 要点 3。
```
